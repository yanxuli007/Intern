    1. Introduction

    App Cockpit is a product from the BMW Group for Developers and Functional Owners of apps for the Online Apps Platform.
    App Cockpit manages Gate 2 of the quality assurance pipeline of apps and releases the apps to the integration and production environment.

    • More informations about App Cockpit

    1.1 Initial situation

    The development team of apps want to test their versions not only on test vehicles but also on real customer cars (their own cars).
    To make this possible they can make specific releases in the AppCockpit that only are effective for specific vins.
    For this to work the cars to test on need to have the necessary services booked.

    With the CASA integration, App Cockpit can add vins to vehicle groups that allow access to those apps.
    This speeds up the testing process for some development teams.

    1.2 Architecture overview

    The figure shows the conception view of the interfaces between CASA and AppCockpit

    1.3 Interface description


    POST/api/vehiclegroups/v2/{vehicleGroupId}/vehiclesAssign VINs to a vehicle groupAssigns a vehicle to a specific Vehicle Group
    POST/api/vehiclegroups/v2/{vehicleGroupId}/vehicles/bulkdeleteRemove VINs from a vehicle groupremoves a vehicle from a specific Vehicle Group

    App Cockpit uses a synchronous REST interface to assign and delete a vehicle from a spcific vehicle group. 

    An asynchronous CASA worflow will be genereated to trigger the provisioning of Connected Drive services.


    The following Vehicle Groups are administrated by App Cockpit:

    • VehicleGroups that are ment to give access to a service for testing purposes.
      This is at most one for each app that wants to use this feature.
    • Currently used Vehicle Groups (19.01.2021): lt3_test_services

    4. Agreement on information- and data privacy

    Applications processing personally identifiable information have to have an appraisal concerning data privacy protection (Prior checking in Germany according to Â§4 Section 5,6 German Federal Data Protection Act (BDSG); in other countries according to their local laws). Therefore, the questionnaire "Data Privacy Application Evaluation" (http://intra.bmwgroup.net/Personal/downloads/Datenschutz/Privacy_Questionnaire.docx) has to be filled out and agreed upon with PD.

    Personally identifiable information may only be stored as long as a contractual or quasi-contractual relationship (examples for quasi-contractual relationships: Applications for employment at BMW AG, offers to customers; examples for contractual relationships: contract of employment, contract for ConnectedDrive, leasing contract) exists or the person has explicitly agreed to the data storage.

    The following agreements have to be concluded within this interface contract. The target application has to proof and confirm respectively:

    4.1 Purpose

    VINs and status change event notifications are exchanged between App Cockpit and CASA in order to enable the functionality of administrating the VINs inside special vehicle groups that enable the cars of app developers and functional owners to install the apps they want to test before submitting them to the release circle.

    4.2 Transfer

    The data exchanged in this interface only serves to enable or disable data transmission from VINs (vehicles) as per the fleet owners' contract requirements. No actual user or energy consumption data is transferred via this interface.

    4.3 Erase

    Vehicle group information gets deleted with the removal of the VIN from App Cockpit.

    Change Management

    Interface Versions

    CASA reserves the right to update the interface to a new version and to deactivate the old interface. The system App Cockpit must change their interface to the new version. This has to happen in a previously defined time period between CASA and App Cockpit.

    Default: App Cockpit must change to the new interface version within the next PI after the switch was done on CASA side.

    Interface changes within one version

    • Incompatible interface changes can only be introduced if all partners agree
    • Compatible changes to an interface can be introduced by CASA
      • Enums may be extended by CASA
      • Optional new fields can be added
