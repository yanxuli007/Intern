    Introduction

    This interface contract deals with the data transfer between system AM-Vehicle and AM-CASA. AM-Vehicle needs to have the updated information for Vehicles from VDWH, this is accomplished by having AM-Vehicle copy the needed data from VDWH and than sending it to AM-Casa for the corresponding update.

    The processes for Vehicle information transfer to AM-Vehicle are described in this document.



    Architecture overview

    AM-Casa receives vehicle messages from AM-Vehicle and sends corresponding response messages about the state (success/error) of the import process on AM-Casa side to AM-Vehicle.

     

    Interface description

    The SQS interface is provided to handle the messages about vehicles from AM-Vehicle, so the information is up to date in the AM-CASA tables. In this process, AM-Vehicle sends the vehicle information to the VehicleExportQueue where AM-CASA is listening for incoming messages, after the messages are processed, AM-CASA sends a response message to AM-Vehicle through the VehicleExportResponseQueue with the result of the operation.

    Interface Event Notification

    Introduction

    Inital situation

    One of the main functionalities of CASA is the provisioning of several target systems concerning the current booked vehicle services of the customer. As so systems like IDS, NOP and MNO Manager are triggered to dis-/enable services or the SIM card in the car. 

    The main purpose of the CASA event stream is to push such events occurred in CASA to other applications. The main information is the event type. Events are vehicle and/or user specific and have one or both attributes. Messages of one Eventtype might originate from different Sources.

    Interface description

    Several events in CASA will lead CASA to send notifications to the generic event interface which is useable by multiple subscribers. Events can be Incoming (from CASA perspective) like "Vehicle has changed" - an Event coming from FBM. Which might lead to a changed set of offerings the next time the user logs in. Events can be Outgoing to other systems, like a "Customer has order an Offer" - an Event going to Buffet, which then takes care about payment processes and forwarding events to subledger/ledger systems in the end. The interface is generic in the sense of that CASA do not filter or limit notifications in any way for a special subscriber. This logic must be implemented on the subscriber’s side if necessary.
